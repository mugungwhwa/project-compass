## Part 1: 서드파티 API 연동 (합법)

### 1.1 핵심 결론

> Compass가 고객의 API Key로 Statsig, AppsFlyer, Adjust 등의 데이터를 조회하는 것은
> **Customer-Authorized Agent 패턴**으로, 법적으로 안전합니다.

| 평가 항목 | 리스크 수준 | 근거 |
|---|---|---|
| 고객 승인 기반 API 접근 | **낮음** | Zapier, Segment, Fivetran 등 수천 개 SaaS의 표준 패턴 |
| 무료 티어 상업 활용 | **낮음** | 무료 API는 사용하라고 제공된 것 |
| 각 플랫폼 ToS 준수 | **낮음-중간** | 플랫폼별 ToS 개별 검토 필요하나 대체로 허용적 |
| 플랫폼 리스크 (ToS 변경) | **중간** | 법적 리스크가 아닌 비즈니스 리스크 |
| 데이터 보호 (GDPR/PIPA) | **낮음-중간** | 표준 DPA 및 컴플라이언스 조치 필요 |

### 1.2 법적 구조

```
Platform ─(계약)─→ Customer ─(위임)─→ Compass

Compass의 접근 권한은 고객의 권리에서 파생됨.
고객이 API Key를 제공하는 행위 자체가 위임의 증거.
```

합법적인 이유:
1. 고객이 데이터 소유자
2. API는 접근하라고 제공된 것
3. 고객의 명시적 위임
4. 변환적 가치 추가 (재판매 아님)

### 1.3 플랫폼별 ToS 분석

| 플랫폼 | API 접근 | 파트너 프로그램 | 리스크 |
|---|---|---|---|
| **Statsig** | 공개 REST API + SDK, 무료 티어 포함 | Amplitude 등과 연동 | **낮음** |
| **AppsFlyer** | Pull/Master/Data Locker API | Technology Partner + Marketplace | **매우 낮음** |
| **Adjust** | KPI Service/Cohort API | 기술 연동 마켓플레이스 | **매우 낮음** |
| **Firebase** | OAuth 기반 (서드파티 위임 설계) | Google Cloud 에코시스템 | **낮음** |
| **LaunchDarkly** | REST API + SDK | 기술 파트너 프로그램 | **낮음** |

### 1.4 유리한 판례

- **Zapier**: 6,000+ 앱 상업 연동, 소송 패소 사례 없음
- **Segment (Twilio)**: 수십 개 플랫폼 데이터 수집/라우팅, 상업 서비스
- **Singular + AppsFlyer/Adjust**: 부분적 경쟁자이면서 API 연동 — Compass의 가장 직접적 선례
- **hiQ v. LinkedIn (2022)**: 비인가 스크래핑도 CFAA 위반 아님 → 인가된 API 접근은 당연히 합법

### 1.5 "무료 티어 유도" 리스크

**리스크: 매우 낮음.** Compass는 Statsig의 성장 파트너:
- 실험 가치를 보여줌 → 고객이 실험 증가 → 무료 한도 초과 → Statsig 유료 전환
- Zapier도 수많은 무료 티어 서비스와 연동하며 상업 운영. 문제 된 적 없음

## Part 2: 앱 인텔리전스 크롤링 (금지)

### 2.1 핵심 결론

> Sensor Tower, AppMagic, data.ai의 유료 콘텐츠를 크롤링하는 것은
> **한국법상 형사 처벌 가능성**이 있으며, 정부 지원사업 수혜 스타트업에게 치명적입니다.

| 크롤링 대상 | 미국법 | 한국법 | 종합 판정 |
|---|---|---|---|
| 앱스토어 공개 페이지 (GP, App Store) | 중간 | 중간 | 주의하면 가능 |
| Apple iTunes Search API (공식) | **낮음** | **낮음** | **권장** |
| Sensor Tower 유료 콘텐츠 (로그인 후) | **매우 높음** | **매우 높음 (형사)** | **절대 금지** |
| 공개 벤치마크 리포트 (GameAnalytics 등) | **낮음** | **낮음** | **권장** |
| 데이터 라이선싱 (계약) | **없음** | **없음** | **가장 안전** |

### 2.2 API 연동 vs 크롤링: 법적 차이

```
API 연동 (합법)                     크롤링 (고위험)
━━━━━━━━━━━━━━━                   ━━━━━━━━━━━━━━━━
플랫폼이 API를 제공 = "들어오세요"   플랫폼이 접근 금지 = "들어오지 마세요"
고객이 자기 데이터를 위임             남의 독점 데이터를 무단 취득
고객의 API Key = 인가된 접근          ToS 위반 접근 = 비인가 (형사 가능)
데이터 소유자 = 고객                  데이터 소유자 = Sensor Tower
```

### 2.3 한국법 리스크 (핵심)

**정보통신망법 제48조** — 형사 처벌 가능
- 대법원 2021도1533 (2022.5.12): "접근 권한"은 기술적 보호조치 + ToS로 객관적 판단
- Sensor Tower: 로그인 필수 + 유료 구독 + ToS 크롤링 금지 → 3중 보호

**부정경쟁방지법 제2조 제1호 카목** (2022년 개정) — 데이터 부정취득 금지
- Sensor Tower의 매출 추정 DB는 보호 대상 데이터 3요건 모두 충족
- 크롤링 = "부정한 수단으로의 데이터 취득"

**저작권법 제93조** — 데이터베이스 제작자의 권리
- 체계적 추출은 "비상당 부분"이라도 "상당 부분"으로 간주

**정부 지원사업 리스크**: 형사 분쟁 연루 시 지원금 환수 + 후속 사업 참여 제한

### 2.4 판례 분석

유리 (공개 데이터에 한정):
- **Van Buren v. US (2021)**: CFAA는 공개 데이터에 적용 불가
- **Meta v. Bright Data (2024)**: 비로그인 공개 페이지 크롤링 허용

불리 (로그인/유료 데이터):
- **hiQ v. LinkedIn (2022)**: 합의 — $50만 배상 + 전체 데이터/코드 삭제
- **Facebook v. Power Ventures (2016)**: 인가 취소 후 접근 → CFAA 위반

→ Sensor Tower는 로그인 + Click-wrap ToS → Meta v. Bright Data 판례가 보호해주지 않음

## Part 3: 공통 한국법 사항

### 3.1 개인정보보호법 (PIPA, 2023년 개정)

| 사항 | Compass 대응 |
|---|---|
| 동의 요건 | 개인정보 처리 시 정보주체 동의 또는 적법 근거 필요 |
| 국외 이전 | 동의 또는 인정된 이전 메커니즘 필요 |
| 처리 위탁 | 고객(컨트롤러)과 Compass(처리자) 간 DPA 필요 |
| 2023년 개정 | 정당한 이익(legitimate interest) 개념 도입 — Compass에 유리 |

```
처리 데이터 유형별 법적 부담:

집계 데이터 (게임 레벨 리텐션, 집계 매출)
  → PIPA 직접 적용 제한적, 법적 부담: 낮음 ✅

유저 레벨 데이터 (개인 리텐션, 개인 LTV)
  → PIPA 전면 적용, 법적 부담: 중간 ⚠️

→ 가능한 한 집계 데이터 처리로 PIPA 부담 최소화
```

### 3.2 데이터산업법 (2022년 시행)

- Compass는 "데이터분석서비스"로 분류 가능
- 법 취지가 데이터 기반 서비스에 **허용적** — Compass에 유리

### 3.3 재무 데이터 보호 (영업비밀)

고객의 재무 데이터(매출, burn rate, 현금 잔액 등)는 **PIPA 개인정보가 아닙니다.** PIPA는 "자연인"에 대한 법률입니다.

그러나 **영업비밀(부정경쟁방지법)**로 보호됩니다:
- 비공개성: 공개되지 않은 회사 재무 정보
- 경제적 유용성: 경쟁사에게 가치 있는 정보
- 비밀 관리성: Compass가 보호 조치를 유지해야 함

| 사항 | Compass 의무 |
|---|---|
| 암호화 | AES-256 at rest + TLS 1.2+ in transit (Supabase 기본 제공) |
| 접근 격리 | Supabase RLS로 테넌트별 완전 격리 |
| 감사 추적 | 재무 데이터 조회/수정/내보내기 전체 로그 기록 |
| 삭제 보장 | 고객 요청 시 모든 재무 데이터 즉시 삭제 가능 |
| 집계 금지 | 개별 고객 재무 데이터를 타 고객에게 노출하거나 벤치마크로 집계하지 않음 |
| 위반 시 | 영업비밀 침해: 형사 10년/벌금 5억 + 민사 손해배상 |

**MSA에 반드시 포함할 조항:**
- Compass는 고객 재무 데이터를 영업비밀로 취급
- 재무 데이터는 인터벤션 효과 평가 및 투자 의사결정 지원에만 사용, 제3자 공유/집계 금지
- 계약 종료 시 30일 이내 완전 삭제

> **포지셔닝 명확화**: Compass는 투자 중개인(funding intermediary)이 아닙니다. 고객의 자본 배분 의사결정을 지원하는 SaaS 플랫폼이며, 투자 자금 조달·중개·자문(금융투자업)을 수행하지 않습니다. 재무 데이터 활용은 고객 내부 의사결정 지원 목적에 한정됩니다.

### 3.4 정보통신망법

- **API 연동**: 고객 인가 API Key 사용 = "비인가 접근" 아님 → 합법
- **크롤링**: 기술적 보호조치 우회 시 형사 처벌 가능

## Part 4: 권장 법적 프레임워크

### 4.1 고객 대상 계약

**A. MSA (마스터 서비스 계약)**
- Compass의 서드파티 데이터 접근 방식 명시
- 고객의 데이터 공유 권한 진술
- API 인증 정보 관리 책임
- 면책 조항 (비인가 데이터 공유 클레임)

**B. 서드파티 연동 약관 (부속서)**
- 지원 연동 목록
- 고객의 서드파티 ToS 검토 확인
- API Key 보안 의무
- 서드파티 API 변경/중단 시 면책

**C. DPA (데이터 처리 계약)**
- GDPR + PIPA 준수 표준 DPA
- 처리 범위, 보안 조치, 침해 통지, 삭제 절차

### 4.2 고객 인가 플로우

```
Step 1: 고객이 "Connect [Platform]" 클릭
Step 2: 고지문 표시 (접근 범위, ToS 확인, DPA 동의)
Step 3: 고객이 "연동 승인" 클릭
Step 4: API Key 입력 또는 OAuth 완료
Step 5: 인가 기록 저장 (타임스탬프 + 감사 추적)
```

### 4.3 내부 컴플라이언스

1. ToS 모니터링 — 각 플랫폼 ToS 변경 정기 검토
2. API Key 보안 — Supabase Vault / GCP Secret Manager
3. Rate Limit 준수 — 자동 rate limiting 구현
4. 데이터 최소화 — 필요한 데이터만 접근/저장
5. 감사 추적 — 모든 API 접근 로그 기록
6. 사고 대응 계획 — C&D 수신 시 즉시 대응 절차

## Part 6: 합법적 마켓 인텔리전스 대안 전략

### 절대 하면 안 되는 것
1. ❌ Sensor Tower / AppMagic / data.ai 유료 콘텐츠 크롤링
2. ❌ 이 플랫폼에 계정 생성 후 ToS 위반 자동 접근
3. ❌ 제3자 스크래핑 서비스를 통한 우회
4. ❌ 매출 추정치를 복사하여 벤치마크 DB 구축

### 안전하게 할 수 있는 것
1. ✅ 공식 앱스토어 API (Apple RSS, iTunes Search) 사용
2. ✅ 공개 벤치마크 리포트 (GameAnalytics, AppsFlyer, Unity) 활용
3. ✅ 공개 앱스토어 페이지 신중한 크롤링
4. ✅ 개발사 데이터 파트너십으로 1차 데이터 확보
5. ✅ 자체 추정 모델 구축 (공개 신호 + 파트너 데이터)
6. ✅ 플랫폼 성장 후 라이선싱 계약 체결

### 단계별 전략

| Phase | 전략 | 비용 |
|---|---|---|
| **Phase 1** (Month 1-6) | 공개 벤치마크 리포트 + 앱스토어 API | $0 |
| **Phase 2** (Month 6-12) | 자체 추정 모델 + 고객 데이터 파트너십 | 개발 비용 |
| **Phase 3** (Month 12+) | 플랫폼 성장 → 앱 인텔리전스 파트너십/라이선싱 | 협상 |

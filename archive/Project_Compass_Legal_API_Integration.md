## 1. 핵심 질문

> Compass(유료 SaaS)가 고객에게 "Statsig 무료 티어에 가입하고 API Key를 주세요,
> 우리가 실험 데이터를 가져와서 투자 가치로 번역해드립니다"라고 하는 것이
> 법적으로 안전한가?

### 결론 요약

| 평가 항목 | 리스크 수준 | 근거 |
|---|---|---|
| 고객 승인 기반 API 접근 | **낮음** | Zapier, Segment, Fivetran 등 수천 개 SaaS의 표준 패턴 |
| 무료 티어 상업 활용 | **낮음** | 무료 API는 사용하라고 제공된 것; 부가가치 서비스 금지 아님 |
| 각 플랫폼 ToS 준수 | **낮음-중간** | 플랫폼별 ToS 개별 검토 필요하나, 대체로 허용적 |
| 플랫폼 리스크 (ToS 변경) | **중간** | 법적 리스크가 아닌 비즈니스 리스크 |
| 데이터 보호 (GDPR/PIPA) | **낮음-중간** | 표준 DPA 및 컴플라이언스 조치 필요 |
| 경쟁 서비스 주장 | **매우 낮음** | Compass는 어떤 연동 플랫폼과도 경쟁하지 않음 |
| 데이터 재판매 주장 | **매우 낮음** | Compass는 데이터를 변환(Transform)하지 재판매하지 않음 |

## 2. 법적 구조: Customer-Authorized Agent 패턴

### 2.1 Compass가 사용하는 법적 모델

```
┌──────────────────┐         ┌──────────────┐         ┌──────────────┐
│  Platform         │◄──────►│  Customer     │◄──────►│  Compass      │
│  (Statsig 등)     │ 이용약관 │  (게임사)     │  서비스  │  (Decision OS)│
│                   │ 계약    │              │  계약   │              │
│  데이터 보관자     │         │  데이터 소유자 │         │  위임된 대리인 │
└──────────────────┘         └──────────────┘         └──────────────┘

법적 권리 체인:
  Platform ─(계약)─→ Customer ─(위임)─→ Compass

Compass의 접근 권한은 고객의 권리에서 파생됨.
Compass는 플랫폼과 직접 계약 관계가 없어도 됨.
고객이 자신의 API Key를 제공하는 행위 자체가 위임의 증거.
```

### 2.2 이 모델이 합법적인 이유

1. **고객이 데이터 소유자** — MMP/실험 플랫폼의 데이터는 고객(게임사)의 데이터
2. **API는 접근하라고 제공된 것** — 플랫폼이 공개 API를 제공한 것 자체가 프로그래밍적 접근을 허용한 것
3. **고객의 명시적 위임** — 고객이 자발적으로 API Key를 Compass에 제공
4. **변환적 가치 추가** — Compass는 원본 데이터를 재판매하지 않고, 투자 결정이라는 새로운 가치로 변환

## 3. 플랫폼별 ToS 분석

### 3.1 Statsig

| 항목 | 분석 |
|---|---|
| **API 접근** | 공개 REST API + SDK 제공, 프로그래밍적 접근이 핵심 기능 |
| **무료 티어** | API 접근 포함 — 기능 제한이지 사용 금지가 아님 |
| **서드파티 접근** | Statsig 자체가 타 플랫폼(Amplitude 등)과 연동, 에코시스템 연동은 예상된 용법 |
| **주의 사항** | Rate limit 준수 필수, 자동화 접근 관련 조항 확인 필요 |
| **리스크** | **낮음** — Statsig의 비즈니스 모델이 에코시스템 연동으로 성장 |

### 3.2 AppsFlyer

| 항목 | 분석 |
|---|---|
| **API 접근** | Pull API, Master API, Data Locker 등 광범위한 공개 API |
| **파트너 프로그램** | **Technology Partner** 프로그램 + Integration Marketplace 공식 운영 |
| **데이터 소유권** | 고객이 데이터 소유자, AppsFlyer는 데이터 처리자 (DPA 명시) |
| **서드파티 접근** | ToS에서 고객이 서비스 제공자에게 데이터 공유 허용 |
| **리스크** | **매우 낮음** — Singular가 이미 AppsFlyer API를 상업적으로 연동 |

### 3.3 Adjust

| 항목 | 분석 |
|---|---|
| **API 접근** | KPI Service API, Cohort API 등 공개 API |
| **파트너 에코시스템** | 기술 연동 마켓플레이스 운영 |
| **리스크** | **매우 낮음** — AppsFlyer와 동일한 에코시스템 패턴 |

### 3.4 Firebase (Google)

| 항목 | 분석 |
|---|---|
| **API 접근** | OAuth 기반 인증 — 서드파티 위임을 위해 설계된 프로토콜 |
| **Google Cloud ToS** | 서드파티 앱을 통한 API 접근을 명시적으로 허용 (고객 인가 조건) |
| **경쟁 서비스 금지 조항** | "substantially similar" 경쟁 서비스 생성 금지 — Compass는 Firebase와 경쟁하지 않음 |
| **리스크** | **낮음** — Google 전체 클라우드 비즈니스가 에코시스템 연동에 의존 |

### 3.5 LaunchDarkly

| 항목 | 분석 |
|---|---|
| **API 접근** | REST API + SDK 제공 |
| **파트너 프로그램** | 기술 파트너 프로그램 + Integration 마켓플레이스 |
| **리스크** | **낮음** — 표준 연동 패턴 |

### 3.6 공통 ToS 제한 사항

모든 플랫폼에 공통적으로 존재하는 제한:

| 제한 사항 | Compass 해당 여부 | 대응 |
|---|---|---|
| Rate Limit 준수 | 해당 | 자동 rate limiting 구현 |
| 경쟁 서비스 금지 | **비해당** | Compass는 어떤 플랫폼과도 직접 경쟁하지 않음 |
| 데이터 재판매 금지 | **비해당** | 원본 데이터 재판매가 아닌 변환적 가치 추가 |
| 법률 준수 | 해당 | GDPR/PIPA 컴플라이언스 필요 |
| 보안 우회 금지 | **비해당** | 공식 API + 고객 인증 정보 사용 |

## 4. 법적 선례 및 판례

### 4.1 유리한 선례

#### Zapier 모델 (6,000+ 앱 상업 연동)
- Zapier는 고객의 API Key/OAuth로 6,000개 이상의 앱에 연결
- 이 연동 서비스 자체를 유료로 판매
- **어떤 플랫폼도 Zapier를 이 모델로 소송하여 이긴 적 없음**
- 많은 플랫폼이 Zapier와 공식 연동, 일부는 비공식이지만 모두 운영 중

#### Segment (Twilio) 모델
- 고객 데이터를 수십 개 플랫폼에서 수집하여 라우팅
- 고객이 API 인증 정보 제공
- 상업 서비스로 과금
- **직접적으로 Compass와 동일한 패턴**

#### Singular + AppsFlyer/Adjust
- Singular는 AppsFlyer/Adjust의 부분적 경쟁자이면서도 API 연동
- 고객이 AppsFlyer/Adjust 계정을 Singular에 연결
- **Compass의 가장 직접적인 선례** — 같은 모바일 게임 업계

#### hiQ Labs v. LinkedIn (2022, 미국 제9순회법원)
- 비인가 공개 데이터 스크래핑도 CFAA 위반이 아니라는 판결
- Compass의 상황은 **훨씬 더 강력**: (a) 인가된 API 접근, (b) 고객의 명시적 위임, (c) 공식 API 사용
- "비인가 스크래핑도 합법이면, 인가된 API 접근은 당연히 합법"

### 4.2 주의해야 할 선례

#### Facebook v. Power Ventures (2016)
- Facebook이 접근 권한을 명시적으로 취소한 후에도 계속 접근
- **교훈**: 플랫폼이 접근을 명시적으로 차단하면 즉시 중단해야 함

#### Twitter/X API 변경 (2023)
- 무료 API 접근을 제한하고 유료화
- 많은 서드파티 앱이 운영 중단
- **교훈**: 이것은 법적 리스크가 아닌 **비즈니스 리스크**. 소송 없이 API 조건 변경으로 발생

#### Reddit API 가격 변경 (2023)
- API 가격 도입으로 서드파티 앱 대량 중단
- **교훈**: 동일 — 비즈니스 리스크. 대응 전략 필요

## 5. 한국법 관련 사항

### 5.1 정보통신망법 (정보통신망 이용촉진 및 정보보호 등에 관한 법률)

| 조항 | 분석 |
|---|---|
| **제49조 (정보의 훼손 등 금지)** | 정보통신망을 통한 비인가 접근 금지. 그러나 고객이 인가한 API Key를 사용한 접근은 "비인가 접근"이 아님 |
| **제28조의2 (개인정보 수집)** | 한국 이용자의 개인정보 처리 시 준수 필요 |

### 5.2 개인정보보호법 (PIPA, 2023년 대폭 개정)

| 사항 | Compass 대응 |
|---|---|
| **동의 요건** | 개인정보 처리 시 정보주체 동의 또는 적법 근거 필요 |
| **국외 이전** | 개인정보 국외 이전 시 동의 또는 인정된 이전 메커니즘 필요 |
| **처리 위탁** | 고객(컨트롤러)과 Compass(처리자) 간 데이터 처리 계약 필요 |
| **2023년 개정** | GDPR과 유사한 정당한 이익(legitimate interest) 개념 도입 — Compass에 유리 |

### 5.3 실무적 함의

```
Compass가 처리하는 데이터 유형에 따른 법적 부담:

집계 데이터 (게임 레벨 리텐션, 집계 매출)
  → PIPA 직접 적용 제한적
  → 개인정보가 아닌 통계 데이터
  → 법적 부담: 낮음 ✅

유저 레벨 데이터 (개인 리텐션, 개인 LTV)
  → PIPA 전면 적용
  → 적법 근거, 동의, 국외 이전 보호조치 필요
  → 법적 부담: 중간 ⚠️

권장: 가능한 한 집계 데이터로 처리하여 PIPA 부담 최소화
     유저 레벨 데이터 필요 시 적절한 법적 기반 확보
```

### 5.4 데이터산업법 (2022년 시행)

- 데이터 활용 및 유통 촉진을 위한 법률
- Compass는 "데이터분석서비스"로 분류 가능
- 법 취지가 데이터 기반 서비스에 **허용적** — Compass에 유리

## 6. "무료 티어 유도" 관련 리스크 분석

### 6.1 우려 사항

> "Compass가 고객에게 Statsig 무료 티어를 사용하라고 권유하는 것이
> Statsig의 비즈니스를 해치는 것은 아닌가?"

### 6.2 분석

| 관점 | 분석 |
|---|---|
| **무료 티어의 목적** | 무료 티어는 사용자 확보를 위해 플랫폼이 자발적으로 제공. 사용하라고 만든 것 |
| **업셀링 효과** | Compass를 통해 실험의 투자 가치를 경험한 고객은 실험을 더 많이 하게 됨 → Statsig 유료 전환 촉진 |
| **Compass의 중립성** | Compass는 특정 티어를 강요하지 않음. 무료든 유료든 모든 Statsig 플랜과 연동 |
| **선례** | Zapier는 수많은 무료 티어 서비스와 연동하며 상업 서비스 운영. 문제 된 적 없음 |

### 6.3 결론

**리스크: 매우 낮음**

오히려 Compass는 Statsig의 **성장 파트너**에 가깝습니다:
- Compass가 실험 데이터의 투자 가치를 보여줌
- 고객이 실험의 가치를 체감 → 더 많은 실험 실행
- 무료 티어 한도(2M events/월) 초과 → Statsig 유료 전환
- **Win-Win 관계**

## 7. 권장 법적 프레임워크

### 7.1 고객 대상 계약

#### A. 마스터 서비스 계약 (MSA) 포함 사항

```
필수 조항:
□ Compass가 서드파티 데이터에 접근하는 방식 명시
□ 고객이 데이터 공유 권한이 있다는 진술
□ 고객의 API 인증 정보 유지 관리 책임
□ 면책 조항 (비인가 데이터 공유로 인한 클레임에 대해 고객이 Compass 면책)
□ 데이터 처리 조건 (GDPR/PIPA 준수)
```

#### B. 서드파티 연동 약관 (별도 부속서)

```
필수 조항:
□ 지원 연동 목록
□ 고객이 서드파티 플랫폼 ToS를 검토했다는 확인
□ API 인증 정보를 Compass에 공유하는 것이 서드파티 계약상 허용됨을 고객이 진술
□ Compass의 API 인증 정보 보안 의무
□ 서드파티 API 변경/중단 시 Compass 면책
□ 플랫폼 ToS 변경 시 처리 절차
```

#### C. 데이터 처리 계약 (DPA)

```
GDPR + PIPA 준수 표준 DPA:
□ 처리 범위 및 목적
□ 처리되는 데이터 카테고리
□ 보안 조치
□ 위탁 처리자 관리
□ 정보주체 권리 처리 절차
□ 침해 통지 절차
□ 계약 종료 시 데이터 삭제
```

### 7.2 고객 인가 플로우

```
Step 1: 고객이 Compass에서 "Connect Statsig" 클릭

Step 2: Compass가 고지문 표시:
        ┌─────────────────────────────────────────────────┐
        │  Statsig 계정 연동                               │
        │                                                  │
        │  연동 시 Compass가 귀하의 Statsig 데이터에        │
        │  접근하여 실험 결과를 투자 가치로 분석합니다.      │
        │                                                  │
        │  귀하는 다음을 확인합니다:                        │
        │  ☑ 본인이 이 접근을 승인할 권한이 있음            │
        │  ☑ Statsig 이용약관에서 서드파티 접근이 허용됨    │
        │  ☑ Compass 개인정보처리방침 및 DPA에 동의         │
        │                                                  │
        │  [연동 승인]  [취소]                              │
        └─────────────────────────────────────────────────┘

Step 3: 고객이 "연동 승인" 클릭

Step 4: 고객이 API Key 입력 또는 OAuth 플로우 완료

Step 5: 인가 기록 저장 (타임스탬프 + 감사 추적)
```

### 7.3 플랫폼 대상 전략

| 전략 | 우선순위 | 설명 |
|---|---|---|
| **기술 파트너 프로그램 가입** | 높음 | AppsFlyer, Statsig 파트너 등록 → 공식 인가 + 마켓플레이스 노출 |
| **공식 연동 등록** | 중간 | 각 플랫폼 Integration 마켓플레이스 등재 |
| **직접 계약** | 낮음 (필수 아님) | 대부분의 경우 고객-플랫폼 계약으로 충분 |

### 7.4 내부 컴플라이언스

```
필수 조치:
1. ToS 모니터링     — 각 연동 플랫폼 ToS 변경 정기 검토
2. API Key 보안     — 암호화 저장 (Supabase Vault / GCP Secret Manager)
3. Rate Limit 준수  — 자동 rate limiting 구현
4. 데이터 최소화    — 필요한 데이터만 접근/저장
5. 감사 추적        — 모든 API 접근 로그 기록
6. 사고 대응 계획   — 플랫폼 클레임/C&D 수신 시 즉시 대응 절차
```

## 9. 최종 결론

### 법적으로 안전한가?

**예, 기본적으로 안전합니다.** 단, 아래 조건을 충족해야 합니다:

1. ✅ 고객의 명시적 인가를 받고 기록할 것
2. ✅ 각 플랫폼 ToS의 rate limit과 사용 제한을 준수할 것
3. ✅ 원본 데이터를 재판매하지 않고 변환적 가치를 추가할 것
4. ✅ 적절한 데이터 처리 계약(DPA)을 체결할 것
5. ✅ API 인증 정보를 안전하게 관리할 것
6. ⚠️ MVP 전에 IT 전문 변호사의 ToS 검토를 받을 것 (비용 대비 리스크 감소 효과 큼)

### 비즈니스 리스크 vs 법적 리스크

```
법적 리스크:  낮음 ████░░░░░░ — 표준 패턴, 선례 풍부
비즈니스 리스크: 중간 ██████░░░░ — API 조건 변경 가능성

→ 법적 리스크보다 비즈니스 리스크 대비가 더 중요
→ 복수 플랫폼 지원 + 직접 데이터 임포트 fallback으로 완화
```

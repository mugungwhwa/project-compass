## 핵심 결론: 크롤링하면 안 됩니다

> **Sensor Tower, AppMagic, data.ai의 유료 콘텐츠를 크롤링하는 것은
> 한국법상 형사 처벌 가능성이 있으며, 정부 지원사업을 받는 스타트업에게
> 치명적인 리스크입니다. 대안 전략을 사용해야 합니다.**

### 한눈에 보는 리스크 매트릭스

| 크롤링 대상 | 미국법 리스크 | 한국법 리스크 | 종합 판정 |
|---|---|---|---|
| 앱스토어 공개 페이지 (Google Play, App Store) | 중간 | 중간 | 주의하면 가능 |
| Apple iTunes Search API (공식) | **낮음** | **낮음** | **권장** |
| Sensor Tower 공개 페이지 (비로그인) | 중간-높음 | 중간-높음 | 비권장 (유용 데이터 거의 없음) |
| Sensor Tower 유료 콘텐츠 (로그인 후) | **매우 높음** | **매우 높음 (형사)** | **절대 금지** |
| 공개 벤치마크 리포트 (GameAnalytics 등) | **낮음** | **낮음** | **권장** |
| 데이터 라이선싱 (Sensor Tower API 계약) | **없음** | **없음** | **가장 안전** |

## 1. API 연동 vs 크롤링: 왜 법적으로 다른가

```
API 연동 (합법)                          크롤링 (고위험)
━━━━━━━━━━━━━━━                        ━━━━━━━━━━━━━━━━
플랫폼이 API를 제공함                    플랫폼이 접근을 금지함
= "들어오세요"                           = "들어오지 마세요"

고객이 자기 데이터를 위임                 남의 독점 데이터를 무단 취득
= 정당한 권한                            = 권한 없음

고객의 API Key 사용                      계정 생성 후 ToS 위반 접근
= 인가된 접근                            = 비인가 접근 (형사 가능)

데이터 소유자 = 고객                     데이터 소유자 = Sensor Tower
= 고객 데이터 활용                       = 타인 재산 침해
```

## 2. 한국법에서의 크롤링 리스크 (가장 중요)

### 2.1 정보통신망법 제48조 — 형사 처벌 가능

**정보통신망 침해행위 등의 금지**: 정당한 접근 권한 없이 정보통신망에 침입하는 행위 금지

- 대법원 2021도1533 판결 (2022.5.12): "접근 권한"의 존재 여부는 **기술적 보호조치의 유무**와 **이용약관의 접근 범위 규정**으로 객관적으로 판단
- Sensor Tower는 **로그인 필수 + 유료 구독 + ToS에 크롤링 명시 금지** → 세 가지 보호조치 모두 존재
- 이를 우회하여 크롤링하면 **형사 처벌** 대상이 될 수 있음

### 2.2 부정경쟁방지법 제2조 제1호 카목 (2022년 개정) — 핵심 조항

**데이터 부정취득 금지**: 2022년 4월 개정으로 신설된 조항

보호 대상 데이터 요건:
- ✅ 사업 거래를 위해 생성/축적된 데이터
- ✅ 전자적 수단으로 상당량 관리
- ✅ 기술적 또는 영업적 정보 (영업비밀 제외)

**Sensor Tower의 매출/다운로드 추정 데이터베이스는 이 세 가지를 모두 충족합니다.**

금지되는 행위:
1. 절취, 사기, 부정접근 등 부정한 수단으로 데이터 취득
2. 정당한 접근 권한자가 부정한 이익을 위해 데이터 유용
3. 부정 취득된 데이터임을 알면서 취득

→ Sensor Tower 유료 계정으로 로그인하여 ToS를 위반하고 대량 크롤링하는 것은 **"부정한 수단으로의 데이터 취득"**에 해당할 가능성이 높음

### 2.3 저작권법 제93조 — 데이터베이스 제작자의 권리

- 데이터베이스 제작자는 데이터베이스의 전부 또는 상당 부분을 복제/배포/방송/전송할 배타적 권리 보유
- "비상당 부분"이라도 **반복적·체계적으로** 추출하여 데이터베이스의 정상적 이용과 충돌하면 "상당 부분"으로 간주
- Sensor Tower의 장르별 매출 추정치를 체계적으로 크롤링하여 벤치마크 DB를 구축하는 것은 이 조항 위반

### 2.4 정부 지원사업 관련 리스크

```
예비창업패키지 등 정부 지원사업 수혜 기업이
형사 처벌이나 법적 분쟁에 연루되면:

→ 지원금 환수 가능성
→ 후속 지원사업 참여 제한
→ 기업 신뢰도 훼손
→ 투자 유치에 치명적

∴ 법적 리스크가 있는 크롤링은 절대 불가
```

## 3. 미국법 판례 분석

### 3.1 유리한 판례 (공개 데이터에 한정)

| 판례 | 연도 | 결과 | 핵심 |
|---|---|---|---|
| **Van Buren v. US** | 2021 | 피고 승 | CFAA는 "완전히 접근 금지된 영역"에만 적용. 공개 데이터에는 적용 불가 |
| **Meta v. Bright Data** | 2024 | Bright Data 승 | **비로그인 상태**의 공개 페이지 크롤링은 ToS 위반 아님 |
| **X Corp v. Bright Data** | 2024 | Bright Data 승 | 비로그인 공개 데이터 크롤링 허용 |

### 3.2 불리한 판례 (로그인/유료 데이터)

| 판례 | 연도 | 결과 | 핵심 |
|---|---|---|---|
| **hiQ v. LinkedIn** | 2022 | 합의 — hiQ가 $50만 배상 + 전체 데이터/코드 삭제 | ToS 위반 계약 위반(breach of contract)은 유효 |
| **Facebook v. Power Ventures** | 2016 | Facebook 승 | 인가 취소 후 계속 접근 → CFAA 위반 |
| **온라인 여행 예약 사건 (Delaware)** | 2024 | 원고 승 | **기술적 접근 제어를 우회**한 크롤링은 CFAA 위반 |

### 3.3 핵심 구분선

```
Meta v. Bright Data가 확립한 핵심 기준:

  비로그인 + 공개 데이터 → 크롤링 가능 (ToS는 "방문자"를 구속하지 못함)
  로그인 + Click-wrap ToS 동의 → 크롤링 시 계약 위반

Sensor Tower / AppMagic의 핵심 데이터는:
  → 로그인 필수
  → Click-wrap ToS 동의 필수
  → 유료 구독 필요
  → ToS에 크롤링 명시 금지

∴ Meta v. Bright Data 판례가 보호해주지 않음
```

### 5.2 Phase 2: 자체 추정 모델 구축 (중기 전략)

Sensor Tower, Apptopia가 실제로 비즈니스를 구축한 방법과 동일:

```
Step 1: 공개 신호 수집
  • 앱스토어 랭킹 (공개) — 시간별 순위 변동
  • 평점 수 변화량 — 신규 설치의 프록시 신호
  • 리뷰 속도 — 참여도의 프록시 신호

Step 2: 1차 데이터 파트너십
  • 게임 개발사와 데이터 공유 계약
  • "귀사의 실제 수치를 (익명화하여) 공유해주시면,
    장르 벤치마크 리포트를 무료로 제공합니다"
  • Apptopia 모델: 125,000+ 앱의 실제 데이터로 추정 모델 학습

Step 3: 추정 모델 학습
  • 공개 랭킹 신호(feature) + 실제 매출/다운로드(label) = ML 모델
  • 자체 추정치 생성 → Compass 독자적 데이터 자산

이 접근이 법적으로 가장 깨끗하며,
장기적으로 Compass의 경쟁 우위(moat)가 됩니다.
```

## 7. 결론

### 절대 하면 안 되는 것

1. ❌ Sensor Tower / AppMagic / data.ai **유료 콘텐츠 크롤링**
2. ❌ 이 플랫폼에 계정 생성 후 **ToS 위반 자동 접근**
3. ❌ Bright Data 등 **제3자 스크래핑 서비스를 통한 우회** (본인 책임 면제 안 됨)
4. ❌ 이 플랫폼의 **매출 추정치를 복사하여 벤치마크 DB 구축**

### 안전하게 할 수 있는 것

1. ✅ **공식 앱스토어 API** (Apple RSS, iTunes Search) 사용
2. ✅ **공개 벤치마크 리포트** (GameAnalytics, AppsFlyer) 활용
3. ✅ **공개 앱스토어 페이지** 신중한 크롤링 (rate limit 준수, robots.txt 존중)
4. ✅ **개발사 데이터 파트너십** 으로 1차 데이터 확보
5. ✅ **자체 추정 모델** 구축 (공개 신호 + 파트너 데이터)
6. ✅ **라이선싱 계약** 체결 후 공식 API 사용

### 핵심 메시지

```
"남의 데이터를 훔치지 말고, 자기 데이터를 만들어라."

이것이 Sensor Tower, Apptopia, Similarweb이 실제로 한 일이고,
이것이 Compass가 가야 할 길입니다.

공개 벤치마크 리포트로 시작 → 자체 추정 모델 구축 → 라이선싱으로 보완
```

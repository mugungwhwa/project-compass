# Operating Intelligence Terminal 구축 여정

## 1. 이 사업의 본질

이 프로젝트의 본질은 단순한 분석 툴이나 BI 대시보드를 만드는 것이 아니다.  
핵심은 **모바일 게임 사업자가 실제로 투자 가능한 범위와 한계를 지속적으로 판단할 수 있게 해주는 의사결정 OS(Operating System)** 를 만드는 것이다.

기존 시장에는 이미 훌륭한 툴들이 존재한다.

- Sensor Tower / AppMagic: 시장 인텔리전스, 경쟁작 벤치마크, 라이브옵스/광고/다운로드/매출 추정
- AppsFlyer / Adjust / Singular: MMP, attribution, incrementality, MMM 연계
- Statsig / Optimizely / LaunchDarkly / Firebase: 실험 및 feature flag, A/B 테스트 실행
- Databricks / Snowflake / BigQuery: 데이터 저장 및 모델링 인프라

하지만 이 툴들은 대체로 다음 중 하나에 집중한다.

1. 시장을 보여준다  
2. 측정을 해준다  
3. 실험을 실행해준다  
4. 데이터를 저장해준다

반면 실제 사업 운영자는 다른 질문을 던진다.

- 지금 이 게임에 추가로 자본을 투입해도 되는가?
- 현재 라이브옵스 개입이 LTV에 어떤 영향을 주는가?
- 실험 속도와 실험 승률은 실제 사업 경쟁력으로 이어지는가?
- 적자를 감수하면서도 더 밀어붙여야 하는가, 아니면 멈춰야 하는가?
- 시장 포화도와 내부 성과를 함께 봤을 때, 지금이 투자 확대 구간인가?

즉 이 제품은 **분석 툴이 아니라 투자판단 번역기** 여야 한다.

---

## 2. 왜 이 문제가 필요한가

모바일 게임 산업은 구조적으로 선투자형 산업이다.

- 개발비 선투입
- UA 비용 선투입
- 라이브 운영비 선투입
- 성과 회수는 장기간에 걸쳐 이루어짐

이 구조 때문에 게임 회사는 항상 아래 딜레마를 겪는다.

- 적자를 어디까지 감수할 것인가?
- 좋은 지표처럼 보이는 것이 실제로 투자 가치가 있는가?
- 시장 벤치마크는 좋아 보여도 우리 팀의 운영 생산성이 따라오는가?
- 실험을 많이 돌리는 것이 정말 회수 가능성으로 이어지는가?

현재 대부분의 데이터 환경에서는 다음이 분리되어 있다.

- 외부 시장 인텔리전스
- 내부 마케팅 성과
- A/B 테스트 결과
- 재무 모델링
- 장기 LTV / Payback / IRR 판단

따라서 사업자는 전체를 한 번에 보지 못한다.  
이 프로젝트는 이 단절을 메우는 것을 목표로 한다.

## 3. 제품 포지셔닝

### 한 줄 정의

**Market Intelligence를 Capital Allocation Intelligence로 바꾸는 게임 산업용 Operating Intelligence Terminal**

### 제품이 하는 일

이 제품은 다음 데이터를 결합한다.

- 외부 데이터: Sensor Tower, AppMagic 등에서 얻는 시장/경쟁/장르/광고/리텐션 추정 정보
- 내부 데이터: 매출, 코호트, UA, ROAS, retention, live ops, 실험 결과, 배포 로그
- 재무 데이터: CAC, gross margin, payback, BEP, IRR, 투자 가능 예산
- 모델 데이터: Bayesian 업데이트, LTV forecast, saturation signal, risk score

그리고 이를 아래 질문에 답하는 구조로 변환한다.

- 지금 추가 투자 가능한가?
- 어떤 액션이 실제 사업 가치를 높이고 있는가?
- 어떤 실험 체계가 조직의 경쟁력이 되고 있는가?
- 현재의 적자인수는 허용 가능한가?
- 시장 상황 대비 내부 속도는 충분한가?

### 제품이 하지 않는 일

- 순수 시장 데이터 판매
- MMP 자체 대체
- 풀스택 A/B 테스트 SDK 엔진 직접 대체
- 일반적인 BI만 제공하는 툴

즉 이 제품은 “측정” 자체보다, **측정 결과를 자본 배분 언어로 번역하는 계층** 이다.

## 4. 핵심 사용자

### 1차 사용자
- 게임사 대표
- 사업개발(BD)
- 마케팅 리드 / UA 리드
- 데이터전략 리드
- 라이브옵스 총괄
- CFO 또는 투자판단 담당자

### 그들이 실제로 원하는 것
- 분석이 좋다는 말
- 예측 모델이 좋다는 말
- 실험을 많이 하고 있다는 말

이 아니라,

- 어디까지 더 투자해도 되는지
- 무엇이 실질적인 성장 기여를 하고 있는지
- 어떤 액션이 손익분기점 도달 확률을 높이는지
- 어떤 실험 운영체계가 장기 LTV를 개선하는지

즉 **분석가용 툴이 아니라 사업 운영자용 툴** 로 설계해야 한다.

## 5. 왜 창업 아이템으로 의미가 있는가

이 사업은 장기적으로 게임회사를 만들기 위한 자금과 기반을 마련하는 별도 사업으로 의미가 있다.

논리는 다음과 같다.

1. 게임을 만들기 전에 게임 사업자들이 실제로 무엇이 불편한지 알고 있다.
2. 현재 시장 데이터 툴의 한계를 직접 느끼고 있다.
3. 분석가가 아니라 사업 운영자 입장에서 필요한 의사결정 레이어를 알고 있다.
4. 이 문제를 해결하면 자체 게임 사업에도 그대로 적용 가능하다.
5. 나중에 게임을 만들 때도 가장 중요한 투자·개입·실험 의사결정 OS를 내부에 이미 가지게 된다.

즉 이 사업은 단순 수익사업이 아니라, 장기적으로 **게임 스튜디오 운영 능력을 선행 확보하는 사업** 이기도 하다.

## 6. 제품 철학: 분석보다 개입

핵심 철학은 명확하다.

**이 플랫폼은 “무슨 일이 일어났는가”를 보여주는 것이 아니라, “어떤 개입이 투자에 어떤 영향을 주는가”를 보여줘야 한다.**

따라서 제품의 중심 화면도 단순 KPI가 아니라 아래 구조여야 한다.

- 개입 액션(UA 증액, Live Ops, 밸런스 조정, 기능 배포, A/B 결과)
- 중간 지표 변화(retention, conversion, payer mix, spend depth)
- 최종 투자 지표 변화(payback, BEP, IRR, capital efficiency)
- 현재 허용 가능 범위와 리스크

즉 “분석 결과”가 아니라 **행동 → 결과 → 투자 판단** 구조로 보여줘야 한다.

## 7. A/B 테스트 플랫폼에 대한 판단

직접 SDK 중심의 실험 플랫폼을 만드는 것은 초기 자원 대비 과도하게 무겁다.

필요한 요소가 많다.

- 클라이언트 SDK
- 서버 할당 로직
- feature flag
- exposure logging
- 이벤트 표준화
- 통계 엔진
- Unity / iOS / Android / 서버 환경 대응
- 배포 안정성 보장

현재 보유 자원(AI 엔지니어링, 마케팅 지식, 데이터 분석/모델링)을 고려하면, 초기에 이것까지 직접 만드는 것은 비효율적이다.

### 따라서 전략은 다음과 같다.

#### 하지 않을 것
- 풀스택 A/B testing SDK 플랫폼 직접 구축
- feature flag 인프라 직접 구현
- Unity/모바일 클라이언트별 실험 런타임 직접 제공

#### 할 것
- 기존 실험 결과를 수집
- 실험 메타데이터를 표준화
- 실험 승률, 배포율, 의사결정 속도, 조직 학습 속도를 계산
- 실험의 투자 기여도를 LTV/Payback/BEP 관점에서 번역

즉 제품은 **A/B 테스트 실행 플랫폼이 아니라, A/B 테스트 투자판단 플랫폼** 으로 가야 한다.

## 8. 자체 데이터 플랫폼 필요성

이 제품은 단순히 외부 데이터를 불러와 시각화하는 수준으로는 차별화가 어렵다.  
자체 모델과 서빙 구조를 갖추려면 내부 데이터 레이어가 필요하다.

### 최소한 필요한 것
- 원천 데이터 적재
- 고객별 데이터 분리
- 공통 스키마 정리
- 모델링 / 피처 엔지니어링
- 예측 결과 저장
- 대시보드/리포트/API 서빙

### Databricks에 대한 판단
Databricks는 좋은 후보이다.  
특히 AI/분석 중심 팀에게 적합하고, ETL → 모델링 → 서빙까지 한 흐름으로 이어가기 쉽다.

하지만 중요한 것은 “Databricks를 쓰느냐”가 아니라,
**우리만의 데이터/모델 운영 레이어를 가지느냐** 이다.

즉 정답은 다음과 같다.

- 자체 데이터 플랫폼은 필요함
- 초기 구현체로 Databricks는 적절함
- 고객에게 Databricks 사용을 강제할 필요는 없음
- 장기적으로는 customer-owned warehouse 또는 hybrid 구조도 고려 가능

## 9. 초기 MVP 방향

초기 MVP는 절대로 모든 것을 다 하려 하면 안 된다.  
핵심은 “투자판단”에 바로 닿는 최소 구조를 만드는 것이다.

### MVP 목표
**실험/마케팅/시장 데이터/재무 모델을 연결해, 추가 투자 가능 여부를 보여주는 화면을 제공한다.**

### MVP 입력 데이터
- Sensor Tower / AppMagic 기반 시장 benchmark
- MMP / 광고 성과 데이터
- 내부 revenue/cohort data
- 실험 결과 테이블 또는 수기 업로드
- 기본 재무 변수(CAC, margin, burn, payback target)

### MVP 핵심 출력
1. 현재 게임/프로젝트의 투자 상태 요약
2. 예상 LTV / payback / BEP trajectory
3. 실험 velocity 및 ship rate
4. 실험/액션별 expected contribution
5. 시장 포화도 대비 내부 성장 가능 구간

### MVP에서 보여줄 핵심 질문
- 지금 더 집행해도 되는가?
- 어떤 액션이 가장 가치 있는가?
- 실험 체계가 실제로 성과를 만들고 있는가?
- 현재 burn을 감수할 이유가 있는가?

## 11. 모델링 방향

이 제품의 모델은 “정답 예측기”라기보다, **투자판단 보조기** 여야 한다.

### 적합한 방향
- Bayesian 업데이트
- 외부 market prior + 내부 실측 posterior 결합
- LTV / retention forecast
- saturation score 예측
- scenario simulation
- uncertainty 포함 예측

### 부적합한 방향
- 외부 추정 데이터를 진실값처럼 사용하는 것
- 복잡한 모델을 만들고 해석 불가능하게 만드는 것
- 지나치게 분석가 중심의 고급 통계 인터페이스

핵심은 **의사결정자가 믿고 행동할 수 있을 정도의 설명 가능성** 이다.

## 13. 고객 검증 순서

### 1단계: 인터뷰
다음 유형의 사람들과 인터뷰
- 모바일 게임 대표
- UA 리드
- 라이브옵스 책임자
- 데이터전략/사업전략 담당자
- 퍼블리셔 BD

질문은 기능이 아니라 아래에 집중
- 어떤 순간에 투자 의사결정이 가장 어렵나?
- 시장 데이터와 내부 데이터가 분리되어 답답했던 적은?
- 실험 결과가 실제 투자 판단으로 이어지지 않는 사례는?
- 현재 가장 알고 싶은 것은 “무엇이 일어났는가”인가, “그래서 어떻게 해야 하는가”인가?

- 시장 데이터 + 내부 데이터 + 실험 결과를 합친 투자판단 리포트
- 실제 의사결정에 도움이 되는지 확인

### 3단계: 대시보드 MVP
반복되는 질문을 구조화해서 제품 형태로 변환

## 15. Sensor Tower / AppMagic과의 관계

이 제품은 그들과 경쟁하려는 것이 아니다.  
오히려 그들의 데이터를 더 가치 있는 의사결정으로 바꾸는 상위 레이어가 되는 것이 이상적이다.

### 경쟁하지 말아야 할 영역
- raw benchmark data
- app market intelligence 자체
- installs / revenue estimate 자체 판매

### 붙어야 할 영역
- 그 데이터를 이용한 capital allocation logic
- 실험/개입과 연결한 투자판단 레이어
- 내부 데이터와 외부 데이터를 결합한 운영 OS

따라서 장기적으로는
- 독립 SaaS로 매출을 만들 수 있고
- 동시에 market intelligence 회사의 인수/제휴 대상도 될 수 있다

---

## 16. 창업자의 강점과 맞는 이유

이 사업은 순수 분석가보다, 오히려 **마케팅 + 사업개발 + 데이터전략 감각** 을 가진 창업자에게 더 잘 맞는다.

이유는 다음과 같다.

- 이 제품의 핵심은 분석 자체가 아니라 사업 번역이다.
- 고객도 대체로 분석가가 아니라 사업 운영자다.
- “좋은 분석”보다 “투자 가능한가”를 묻는다.
- 게임 스튜디오 운영 경험/문제의식이 제품 철학으로 이어질 수 있다.

즉 분석 전문성이 부족한 것이 약점만은 아니다.  
오히려 **현업 운영자가 정말 원하는 질문을 중심에 놓을 수 있다는 점이 강점** 이다.

## 17. 전체 여정 요약

### 단계 1: 문제 명확화
- 시장 데이터는 있는데 투자 판단은 어렵다
- 분석은 많은데 행동 결정은 어렵다
- 실험은 돌아가는데 자본 배분 논리로 연결되지 않는다

### 단계 2: 포지셔닝 확정
- 분석 툴이 아니라 의사결정 OS
- market intelligence를 capital allocation intelligence로 번역

### 단계 3: 구현 범위 절제
- A/B SDK 직접 구축은 하지 않음
- 기존 실행도구 위에 올라가는 decision layer로 감

### 단계 4: 데이터 기반 마련
- 자체 데이터/모델 운영 레이어 구축
- Databricks 등으로 MVP 데이터 플랫폼 마련

### 단계 5: 파일럿 검증
- 수작업 리포트/간이 대시보드로 고객 반응 확인
- 실제 의사결정에 쓰였는지 검증

### 단계 6: SaaS화
- 반복적인 의사결정 흐름을 제품화
- 투자 허용선, action impact, experiment investment intelligence 구현

### 단계 7: 확장 또는 Exit
- 독립 SaaS로 성장
- 또는 Sensor Tower / AppMagic / 인접 enterprise intelligence 회사와 제휴/매각

## 18. 최종 정의

이 제품은 데이터 분석 플랫폼이 아니다.  
이 제품은 **모바일 게임 사업자가 불확실성 속에서도 투자 가능한 범위와 개입의 가치를 지속적으로 판단할 수 있게 해주는 의사결정 운영체계** 이다.

즉,

- 시장을 보는 툴이 아니라,
- 시장과 내부 액션을 연결하고,
- 개입이 사업에 미치는 영향을 시각화하며,
- 적자인수의 한계와 가능성을 보여주는,
- 게임 산업용 Operating Intelligence Terminal 이다.
